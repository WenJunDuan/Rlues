## 用途
模糊需求 → 结构化 Spec + 可评估的候选方案。(SDD spec-first 模式)

## 流程 (发散→收敛)

### 1. 定义问题
用一句话说清这个需求要解决什么。不要复述用户的话, 要提炼本质。

### 2. 发散搜索
不限制方向, 尽可能多地收集信息:
- **动手**: `augment-context-engine` 搜项目代码 — 有没有类似功能已实现?
- **动手**: `mcp-deepwiki` 查候选库文档 — 有没有成熟的库可以直接用?
- **动手**: `cat .ai_state/knowledge.md | grep 关键词` — 项目历史经验
- **动手**: `cat .ai_state/lessons.md` — 上次踩过什么坑?
- **动手**: `WebSearch` — 社区怎么解决类似问题的?

### 3. 追问筛选 (苏格拉底式)
- 为什么用这个库而不是那个? 维护状态如何?
- 数据量是现在的 100 倍, 这个方案还能工作吗?
- 最简单的实现是什么? 能不能更简单?
- 核心假设如果是错的, 影响范围多大?

### 4. 收敛输出 → .ai_state/design.md

**Spec 部分** (SDD 模板):
```markdown
## Spec: {需求标题}
### 功能需求
- [MUST] ...
- [SHOULD] ...
- [COULD] ...
### 非功能需求
- 性能: ...
- 安全: ...
- 兼容性: ...
### 约束
- 技术栈: ... | 时间: ... | 依赖: ...
### 验收标准 (可测试条件)
- [ ] ...
```

**候选方案部分** (2-3 个):
每个包含: 描述 / 优势 / 劣势 / 预估工作量 / 技术风险

### 5. 检查点
cunzhi [DESIGN_DIRECTION] — 用户选择方案后进入 R 阶段
